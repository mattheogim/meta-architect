# Master Plan 자체 평가 — 적대적 리뷰

> 이 문서는 MASTER-PLAN.md의 약점, 모순, 비현실적 부분, 논리적 허점을 찾는다.
> 평가 기준: "이 플랜대로 실행했을 때 실패할 가능성이 가장 높은 곳은 어디인가?"

---

## 🔴 Critical: 플랜이 실패할 수 있는 핵심 문제

### C1. MVP가 없다 — 전부 만들고 나서야 평가

**문제**: Phase 0→1→2→3→4 순서대로 전체를 만들고, Phase 4에서야 "이게 되는가?" 평가.
17개 에이전트 + 전체 인프라를 다 만든 후에 "3-agent보다 나은가?"를 확인.
만약 안 나으면? 2~3주를 버린다.

**비교**: 소프트웨어 개발에서는 MVP(최소 기능 제품)로 가설을 빨리 검증한다.
우리 시스템의 핵심 가설("강제된 인지 다양성이 더 나은 아키텍처를 만든다")은 
에이전트 3개만으로도 검증할 수 있다.

**제안**: 
- Phase 1.5 추가: **3-agent MVP** (Architect + Critic + Synthesizer)로 먼저 Baseline B를 만들고,
  점진적으로 에이전트를 추가하면서 각 추가의 marginal value를 측정
- 이러면 "17개 중 어느 에이전트가 실제로 가치를 추가하는가"도 알 수 있음

**심각도**: 🔴 — 이거 안 하면 2~3주 후 "다시 처음부터" 가능성

---

### C2. 타임라인이 비현실적

**문제**:
| Phase | 플랜 | 실제 필요 | 이유 |
|-------|------|----------|------|
| Phase 2 (프롬프트) | 5~7일 | 2~4주 | 17개 × 500~2000줄 = 최소 8,500줄 프롬프트. 하루 1,200줄? 각 프롬프트가 해당 분야 지식 문서임 |
| Phase 1 (Blackboard) | 3~5일 | 1~2주 | 13슬롯 스키마 + 상태 전이 엔진 + Router 15규칙 + LangGraph 그래프. 복잡한 상태 머신 |
| Phase 4 (평가) | 3일 | 1~2주 | 5 시나리오 × 4 baseline = 20회 실행. 전문가 blind review 3명 섭외+실행 시간 |

**제안**: 
- 타임라인을 2~3배로 늘리거나
- MVP 접근으로 scope를 줄이거나
- 프롬프트를 처음부터 2000줄로 안 쓰고, 핵심 200~500줄로 시작 → 반복 개선

**심각도**: 🔴 — 비현실적 타임라인은 프로젝트를 좌초시키는 1번 원인

---

### C3. 전문가 평가가 플랜에만 있고 실행 방법이 없다

**문제**: "3명의 시니어 아키텍트에게 blind review" — 누구? 어떻게 섭외? 비용은?
이게 현실적인가? 3명이 5개 시나리오 × 4개 baseline = 60개 아키텍처를 각각 평가?

**제안**:
- 전문가 대신 **LLM-as-Judge** (다른 모델)로 대체 가능 — MAJ-Eval 프레임워크 적용
- 또는 **자기 평가 + 1명 전문가**로 축소
- Phase A에서는 정량 지표(coverage, cost)만 사용, 정성 평가는 Phase B로 분리

**심각도**: 🔴 — 평가할 수 없으면 가설을 증명할 수 없음

## 🟡 Major: 논리적 허점

### M1. 논문 근거의 도메인 불일치

**문제**: 우리가 인용한 핵심 논문들은 대부분 **math/reasoning/code** 벤치마크에서 측정됨.
- Du et al.: arithmetic, GSM8K, chess
- Hegazy: GSM-8K (수학)
- Qian: MacNet (코딩 + 추론)
- Han & Zhang: commonsense, reasoning, math

**소프트웨어 아키텍처 설계**는 이 태스크들과 근본적으로 다르다:
- 정답이 없음 (수학은 정답이 있음)
- 주관적 판단이 핵심 (trade-off는 "맞다/틀리다"가 아님)
- 매우 긴 출력 (수학은 숫자 하나, 아키텍처는 문서 전체)

**결론**: 논문 근거가 "이 도메인에서도 작동한다"는 증거가 아니라 "다른 도메인에서 작동했다"는 유추.

**완화**: 이걸 명시적으로 인정하고, Phase 4 평가가 이 gap을 메우는 역할임을 문서화

---

### M2. Multi-Model 전략의 모순 미해결

**문제**: 플랜에서 두 논문을 동시에 인용:
- Hegazy(2024): 다양 모델 91% > 동질 모델 82% → "다른 모델 써라"
- Li(2025): Self-MoA > Mixed-MoA +6.6% → "같은 모델이 낫다"

플랜의 해결: "같은 역할에는 같은 모델, 다른 역할에는 다른 모델"
**이건 우리의 해석이지, 검증된 결론이 아니다.**

**제안**: 
- 평가 Phase에서 **All-Opus vs Mixed-Model**을 명시적으로 비교
- 이 비교 자체가 하나의 실험이 됨

---

### M3. "50라운드" 글로벌 상한의 근거 부족

**문제**: 수렴 연구에서 "7라운드에서 0.892 수렴"인데 왜 50?
7 × 6 Phase ≈ 42로 계산한 건가? 근거가 명시되지 않음.

**제안**: 
- Phase당 최대 라운드(예: 10)를 설정하는 게 글로벌 상한보다 적절
- 또는: "3연속 Phase에서 변화 없으면 종료" 같은 동적 종료 조건

---

### M4. Blackboard "자유 기여" vs Phase 게이트 모순 미해결

**문제**: 리서치에서 이 모순을 발견했는데, 플랜이 이걸 해결하지 않았다.
"의식적으로 유지"에 넣었지만, **어떻게 관리하는지**가 없음.

구체적으로:
- v2는 "에이전트가 자유롭게 기여"라고 하면서 Phase 게이트로 순서를 강제
- 플랜은 LangGraph(DAG)를 쓰겠다고 하면서 Blackboard(자유 기여)를 유지

**이 두 개는 구조적으로 다른 패턴이다.** 어느 쪽을 우선하는가?

**제안**: 
- "Blackboard은 데이터 공유 패턴, Phase는 제어 흐름 패턴"으로 명확히 분리
- LangGraph 그래프 = Phase 제어 흐름
- Pydantic 공유 상태 = Blackboard 데이터 공유
- 이 분리를 v2 설계서에 명시

---

### M5. RAG 파이프라인을 "즉시 구현 가능"이라고 했지만

**문제**: Precedent Researcher의 RAG는 "Semantic Scholar API + GitHub API"로 즉시 된다고 했는데:
- 소프트웨어 아키텍처 패턴에 대한 전문 임베딩 DB가 없음
- Semantic Scholar는 학술 논문. 실무 아키텍처 결정은 기술 블로그/GitHub에 있음
- "이 문제를 누가 이미 풀었는가"를 검색하려면 **도메인 특화 knowledge base가 필요**
- 이건 별도 프로젝트 수준의 작업

**제안**:
- v1에서는 RAG 없이 LLM 내재 지식만 사용 (evidence_level: theoretical로 표시)
- v2에서 RAG 추가 (별도 프로젝트로)
- 또는: web_search 도구를 Precedent Researcher에 직접 연결 (실시간 검색)

---

### M6. Recorder를 "코드(LLM 불필요)"로 분류했지만

**문제**: v2 QOC 문서에서 명시적으로 "Recorder를 이벤트 리스너로 축소하지 않음: QOC/ADR 변환은 해석 작업"이라고 했다.

QOC 변환 예시:
- 새 항목 → Question 추출 + Option 정리 + Criteria 매핑
- conflict 등록 → Counter-argument 추가
- accepted → ADR 생성 (Context 요약 + Decision + Consequences)

이건 단순 로깅이 아니라 **추론이 필요한 변환**. Haiku로 커버되는가?

**제안**: Recorder를 "로깅 = 코드, QOC 변환 = Sonnet"으로 분리 (플랜에서 이미 암시했지만 명시적이지 않음)

## 🟢 Minor: 개선하면 좋은 것

### m1. paper-feature-map.json의 source_files가 아직 존재하지 않는 경로

hook이 `src/agents/simplicity_advocate.py`를 매칭하지만 이 파일은 아직 없음.
→ 파일 생성 시 매칭 되므로 문제 없지만, 경로가 바뀌면 hook이 작동 안 함.
→ 구현하면서 경로 업데이트 필요.

### m2. 개발 중 API 비용 예산이 없음

17개 에이전트 × 개발 중 반복 테스트 = 상당한 API 비용.
예상: Phase 2~3에서만 $50~$200 소요 가능.

### m3. 에러 핸들링/fallback 설계가 없음

LLM API 호출 실패, timeout, rate limit 시 어떻게 하는가?
LangGraph의 retry/fallback 메커니즘을 활용해야 하지만 플랜에 없음.

### m4. 관측성(Observability) 구현 시점이 늦음

Recorder의 observability는 Phase 5(반복 개선)에 있지만,
Phase 3(통합 테스트)에서 이미 필요. 디버깅을 위해 최소 로깅은 Phase 1부터.

## 권장 수정 사항 (우선순위순)

1. 🔴 **MVP-First 접근 추가**: 3-agent MVP → 점진적 에이전트 추가 → marginal value 측정
2. 🔴 **타임라인 현실화**: Phase 2를 2~4주로 늘리거나 프롬프트 scope 축소
3. 🔴 **평가 방법 현실화**: 전문가 → LLM-as-Judge(MAJ-Eval) + 정량 지표 우선
4. 🟡 **Blackboard/Phase 모순 해결**: "데이터 공유 vs 제어 흐름" 명시적 분리
5. 🟡 **RAG scope 축소**: v1은 LLM 지식만, RAG는 별도 후속
6. 🟡 **Multi-model은 실험으로**: All-Opus vs Mixed를 평가 항목에 추가
7. 🟢 **관측성 Phase 1으로 당김**: 최소 로깅/tracing부터
8. 🟢 **API 비용 예산 설정**: 개발 중 $200 cap
